repo-local agent harness

A thin control plane for cold coding agents.

agents-kit installs a portable .agents system into a target repo: router, resolvers, gates, skills, logs, and history. It gives an agent just enough structure to classify work, stay in scope, prove done, and repair repeated misses.

seed

Install the default harness, review the diff, localize it in the target repo.

quarry

Take the control-plane pieces that fit. Do not make this package runtime law.

owned

After install, the target repo owns its .agents files and evolves them locally.

why it exists

Cold agents guess unless the repo gives them a harness.

Most coding agents enter a repo without a local operating surface. They infer task shape from chat, inspect whichever files look nearby, and call work done with whatever proof they remember.

What kind of work is this?

Docs, product code, skill work, repair, or rule quality.

What can it touch?

Owner surface, allowed writes, and forbidden surfaces.

What proves done?

A gate names the real check before the claim.

What happens after a miss?

Classify, repair the owner surface, then verify.

control plane

Small files, separate jobs.

The harness works because each surface has one job. It keeps rules loadable and stops logs, history, skills, and gates from impersonating each other.

request to proof

The harness turns a chat request into a checked path.

1. Request

The user asks for work in natural language.

2. Route

The router picks the resolver, gate, and skill.

3. Receipt

The agent names owner, writes, forbidden surfaces, and first check.

4. Work

Edits stay inside the selected owner surface.

5. Gate

Done means the named proof ran, or the blocker is visible.

before it edits

The quick eval is the point.

For nontrivial work, the agent should say how it understands the task before changing files. If the owner, scope, or first check is wrong, you can correct it while the repo is still untouched.

install paths

Existing repo

Installed repo

repair loop

Repeated misses become harness repairs, not vibes.

Not every mistake becomes doctrine. The repair loop is for repeated misses, weak gates, stale task state, and unclear owners. The fix is the smallest live surface that prevents the same failure next time.

1. Miss

Check failed, correction repeated, scope drifted.

2. Classify

factory-failure names the failure class.

3. Own

Pick router, resolver, gate, skill, script, log, or history.

4. Patch

Change the smallest live surface that prevents the repeat.

5. Verify

Run the owner gate and health check when structure changes.

proof

The seed has checks. Installed repos get checks too.

Source repo

npm run verify

Checks the required template file list and runs the bundled health check against templates/default.

Installed repo

python3 .agents/skills/agents-kit/scripts/check-agents-kit-health.py

Checks router references, resolver shape, gates, skill inventory, retired surfaces, and lesson metadata.

The maintenance rule is blunt: keep AGENTS.md small, keep live law in .agents, keep active state in .scratch, keep dated evidence in history, and let the target repo own its local doctrine.
